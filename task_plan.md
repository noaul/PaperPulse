## 关键问题
1. 如何提升项目复杂度和工程化程度，同时不破坏当前功能。
2. 如何加入 workflow/execution/log 模型，但保持旧接口兼容。
3. Docker 流程必须先本地验证，用户确认后再推送和部署。

## 已做决策
| 决策 | 理由 |
|------|------|
| 前端保留现有页面模式 | 用户明确要求前端采用现在的方式，可适当美化 |
| 第一轮重构采用轻量 workflow，不做完整 n8n 画布 | 控制风险，同时显著提升架构复杂度 |
| Docker 先本地验证，用户确认后再推送 | 用户明确偏好 |
| 长任务分析改后台执行 + 轮询进度 | 前端 axios 30s 超时，不应让 AI 批处理阻塞单个 HTTP 请求 |
| 新增独立“分析结果”页面 | 用户明确要求要有能看到分析结果的页面 |
| 后续优先做“报告中心 + 摘要增强 + 邮件预览/重发” | 直接提升当前论文监控闭环，复杂度可控 |
| WeKnora 联动先做“手工 Markdown 知识同步”，再做搜索/问答 | WeKnora REST API 支持 Markdown 知识创建、知识搜索和知识问答；先同步报告最稳 |
| 分析暂停/取消采用协作式控制 | 已发出的单篇 AI 请求无法强杀，因此在每篇论文前后检查暂停/取消状态，保证任务可控且不破坏数据一致性 |
| 全面升级先做路线图再实现 | 当前已有 workflow、分析页、部署流程；直接大改风险高，需要先分层拆分产品能力、数据模型、任务系统和 UI |
| P0 先实现报告中心和投递记录 | 当前抓取、分析、邮件闭环最缺持久化可见性；保存报告后刷新页面不丢失，且能追踪发送状态 |
| 前端重构采用统一全局主题优先 | 当前页面模板化问题主要来自基础视觉系统；先统一背景、卡片、按钮和应用外壳，可低风险提升全站观感 |
| 阅读队列作为独立轻量工具接入 | PaperPulse 已有自动抓取论文库；手工阅读队列应服务于临时收藏/外部文章，不强行混入 RSS Paper 模型 |
